# 知识候选池 — Chat → Wiki 晋升管线

## 一句话

聊天与智慧提炼产生**可晋升 Wiki 的候选事实**，先入池、再解析、最后编译进企业 Wiki；与 `memories`（Chat 召回）双轨并行。

---

## 用户如何感知（前端 UX）

后台管线对用户**默认不可见**，需在 Chat / 概览两处给反馈：

| 触点 | 组件 | 用户看到什么 |
|------|------|-------------|
| **Chat 侧栏底部** | `chat-evolution-panel` | 四阶段管线（对话→提炼→候选→Wiki）+ 本会话近期动态 |
| **每轮回答下方** | `TurnEvolutionHint` | 「正在提炼…」→「智慧已沉淀 / 进入 Wiki 候选」 |
| **删除对话弹窗** | `chat-delete-session-dialog` | L2 复盘确认 → 加载 → 摘要 / Wiki 建议 / 冲突结果 |
| **知识管理概览** | `knowledge-pipeline-banner` | 静态管线说明 |
| **概览近期动态** | `recent_activity` | 新增 `kind=candidate` 条目 |
| **待晋升 Wiki** | `candidate-queue` | 可操作队列 |

### API

```
GET /api/v1/orgs/{org}/chat/sessions/{session_id}/pipeline
```

返回 `stages`（idle/active/done）、`stats`、`recent`（memory + candidate 事件）。

Chat 页在发送消息后轮询该接口约 18s，直到 `event_count` 增加或超时。

### Chat 流式输出（SSE）

```
POST /api/v1/orgs/{org}/chat/stream
```

事件顺序：`status(gathering)` → `status(writing)` → `sources` → `token`×N → `done`（含完整 turn）。  
WebUI 默认走 `/api/kb/{org}/chat/stream` 代理，回答逐字显示；L1 提炼仍在 `done` 后异步触发。

```
Raw Layer          chat_messages / 上传 chunks
    ↓
Candidate Layer    knowledge_candidates（本模块 P0）
    ↓
Resolver Layer     规则匹配 + 冲突标记（P0）→ ingest EntityResolver（P1）
    ↓
Wiki Layer         WikiManager 写 .md（P1 ChatDigestCompiler）
    ↓
Lint Layer         质量检查 / 过期标记（P2）
```

## 与「智慧进化」的关系

| 轨道 | 表 | 用途 | 写入时机 |
|------|-----|------|----------|
| 智慧召回 | `memories` | Chat 上下文、智慧进化页 | L1 每轮 + L2 复盘 |
| Wiki 晋升 | `knowledge_candidates` | 待审核 / 待编译进 Wiki | L1/L2 **双写**（有门槛） |

详见 [4-智慧进化.md](./4-智慧进化.md)。

**晋升门槛**（`candidate_service.should_enqueue_for_wiki`）：
- `preference` → **不进**候选池（仅 memory）
- `project` / `decision` / `workflow` / `rule` / `product` / `methodology` / `entity` → 进池
- 其它类型：`importance ≥ 0.85` 才进池

## API

Base：`http://localhost:8006/api/v1`（WebUI 经 `/api/kb/{orgId}/...` 代理）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/orgs/{org}/candidates?status=pending` | 列表 |
| GET | `/orgs/{org}/candidates/stats` | 统计 |
| POST | `/orgs/{org}/candidates/resolve` | 批量 Resolver（body: `user_id`, `limit`） |
| POST | `/orgs/{org}/candidates/compile` | 将 approved 编译进 Wiki（body: `user_id`, `limit`） |

## Resolver（P0 行为）

`resolve_pending_candidates` 对 `pending` 候选：

1. **Wiki 标题匹配** → `supplement` + `target_wiki_path` + `status=approved`
2. **同标题 memory 已存在** → `update` + `approved`
3. **memory_id 不一致但同标题** → `conflict`
4. **confidence ≥ 0.88 且无匹配** → `create` + `approved`
5. 其它 → 保持 `pending`，`resolver_action=noop`

P0 仅打标；P1 起可写入 Wiki 文件。

```bash

  -d '{"limit": 20}'
```

WebUI：知识管理概览 → **待晋升 Wiki** 面板 →「批量解析」「编译进 Wiki」。

### ingest 双写（P1）

`IngestAgent.run` 在写出 Wiki 后调用 `enqueue_from_ingest_compile`：
- 每条实体/流程/规则 mirror 一条 `knowledge_candidates`
- `status=merged`，`target_wiki_path` 已填，供审计与统一检索
