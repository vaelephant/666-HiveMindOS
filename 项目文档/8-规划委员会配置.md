# 规划委员会配置

> 角色分工外置配置文件：`agent_engine/settings/planning_committee.yaml`  
> 加载代码：`agent_engine/domain/committee_config.py`  
> 编排入口：`agent_engine/agents/planning_committee.py`

---

## 一、职责边界

| 层次 | 配置文件 | 谁维护 | 定义什么 |
|------|----------|--------|----------|
| **委员会结构** | `agent_engine/settings/planning_committee.yaml` | 平台 / 企业管理员 | 有哪些角色、发言顺序、何时触发 |
| **角色怎么说** | `knowledge_base/prompts/prompts.yaml` → `agents.planning_*` | 平台工程 | LLM system / user 模板 |
| **能做什么** | `agent_engine/settings/task_tools.yaml` | 平台工程 | action 白名单 |
| **做得好不好** | `agent_engine/settings/rubrics/*.yaml` | 企业业务 | 评分维度 |
| **何时要人** | `agent_engine/settings/task_gates.yaml` | 企业 + 平台 | gate 策略 |

**原则：** yaml 定「谁出席、按什么顺序开会」；prompts 定「怎么发言」；rubrics / gates 定「按什么标准」；Agent 只能在工具白名单内合成 Plan。

---

## 二、配置文件结构

```yaml
label: 规划委员会
description: ...

trigger_sources:          # 走委员会的任务来源
  - chat_upgrade          # Chat 升级为任务
  - task_center           # 自主任务页直接创建

roles:                    # 发言顺序 = order 升序
  - id: domain            # 稳定 ID，勿随意改
    label: 领域顾问
    description: ...      # UI 展示用
    prompt: planning_domain   # 对应 prompts.yaml 中 agents.planning_domain
    order: 1

  - id: risk
    prompt: planning_risk
    reads_prior: [domain]     # 可读先前角色输出（文档性，编排已内置）

  - id: chair
    prompt: planning_chair
    synthesizes_plan: true    # 标记为合成最终 Plan 的角色
    reads_prior: [domain, risk]

fallback:                 # LLM 失败时的规则兜底
  domain:
    default_summary: ...
  risk:
    by_task_type:
      wiki_organize_decisions: ...
    default: ...
```

---

## 三、三角角色说明

| id | 默认 label | prompt | 职责 |
|----|------------|--------|------|
| `domain` | 领域顾问 | `planning_domain` | 建议步骤、`success_criteria`、可跳过 action |
| `risk` | 风险审查 | `planning_risk` | `estimated_risk`、`gate_overrides` |
| `chair` | 主持人 | `planning_chair` | 合成合法 `tasks[]` Plan JSON |

## 四、如何新增 / 调整角色

### 4.1 只改展示名称或顺序

编辑 `agent_engine/settings/planning_committee.yaml` 中对应角色的 `label`、`description`、`order`，重启服务。

前端会从任务 `plan.committee_roles` 读取（与 yaml 同步写入），无需改 React 代码。

### 4.2 新增第四个角色（示例：法务顾问）

1. 在 `planning_committee.yaml` 的 `roles` 中追加：

```yaml
  - id: legal
    label: 法务顾问
    description: 审查对外输出与合规风险
    prompt: planning_legal
    order: 25
    reads_prior: [domain]
```

2. 在 `prompts/prompts.yaml` 增加 `agents.planning_legal`（system + user_template）。

3. 在 `planning_committee.py` 的 `_llm_round` 中为新角色补充 `kwargs` 组装逻辑（或后续改为通用 `reads_prior` 驱动）。

4. 若需规则兜底，在 `fallback` 节增加对应段落。

> **注意：** `id` 一旦用于生产任务纪要，不宜重命名；`prompt` 名必须与 `prompts.yaml` 键一致。

### 4.3 调整触发范围

修改 `trigger_sources`：

- 去掉 `task_center` → 自主任务页改回单 Planner
- 增加 `api` 等自定义 source → 创建任务时在 `constraints.source` 传入

## 六、热更新

```python
from agent_engine.domain.committee_config import reload_committee_config
reload_committee_config()
```

或重启 uvicorn。修改 yaml 后未 reload 则仍用 `@lru_cache` 中的旧配置。
